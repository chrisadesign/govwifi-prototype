Feature and versioning system guidance

This fork has a versioning system built in, which allows you to create a feature – these can be any structure you want, for example you might organise by hypotheses or sprints – which then has as many versions of that thing as you need.

How the folders are structured

There is an example-feature folder within views. In there is an index page which acts as the changelog for each feature. There is also a v1 folder which has an example question page and a _routes.js file; each version of a feature has it’s own routes file.

Each feature folder sits in the views folder. Versions within a feature must be a number prefixed with a ‘v’, for example ‘v12’.

Routes are controlled per version of a feature, the main routes.js file in views imports the version routes as long as the folder structure is followed.

How to create a new feature

When you start on a new feature all you need to do is dupliacte the example-feature folder and rename it.

When you need a new version you can duplicate the v1 folder and link to it from the index/changelog page.
